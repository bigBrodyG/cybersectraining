EFF — AES-CTR with 2-Byte Per-Block Nonce Brute-Force

January 1, 2025 · 3 min · bigBrodyG
Table of Contents

  • Solution
  • What I Learned

The Challenge#

eff.py implements a non-standard AES-CTR mode. For each flag character at position i, the server constructs a 16-byte plaintext block: 15 bytes of a padding character c followed by the flag character. The key is printed plaintext (you know it). The nonce changes on every block: 2 random bytes prepended to a counter form the 16-byte nonce. The encrypted blocks are printed.

The known key is a gift — the challenge is recovering the 2-byte per-block nonce.

Approach#

Standard AES-CTR uses a fixed nonce; here each block gets a fresh 2-byte random nonce component. That’s only 65536 possibilities per block. I know:

  • The key (given)
  • The plaintext structure: the first 15 bytes of any block are the repeated padding char; only position 15 is the unknown flag character
  • The ciphertext block

For each encrypted block, I brute-force all 65536 possible 2-byte nonces. For each candidate nonce n, I compute AES_ECB_encrypt(key, n || counter) (the keystream block) and XOR with the ciphertext block to get a candidate plaintext. I then check two things: are the first 15 bytes all equal to the padding char c? Is byte 15 a printable ASCII character? If yes, I’ve found the nonce and the flag character.

Because the structure is so rigid — 15 identical bytes followed by one variable byte — there’s almost no false-positive rate. The check is tight enough that at 65536 candidates per block the attack is fast.

from Crypto.Cipher import AES
import struct

def brute_nonce(key, ciphertext_block, pad_char, block_idx):
    """Try all 2-byte nonces for a block. Returns (flag_char, nonce) or None."""
    counter_bytes = struct.pack('>Q', block_idx)  # 8-byte counter (adjust if different)
    for nonce_int in range(65536):
        nonce_bytes = struct.pack('>H', nonce_int)  # 2-byte nonce
        # Construct the 16-byte counter block: nonce || counter (zero-padded to 16)
        ctr_block = nonce_bytes + b'\x00' * 6 + counter_bytes
        keystream_block = AES.new(key, AES.MODE_ECB).encrypt(ctr_block)
        plaintext_block = bytes(keystream_block[i] ^ ciphertext_block[i] for i in range(16))
        if (all(b == pad_char for b in plaintext_block[:15])
                and 0x20 <= plaintext_block[15] <= 0x7e):
            return chr(plaintext_block[15]), nonce_int
    return None, None

# Load key and ciphertext blocks from server output
key = bytes.fromhex("...")          # printed by server
blocks_hex = [...]                  # list of 32-char hex strings from server

flag = ""
for i, block_hex in enumerate(blocks_hex):
    ct_block = bytes.fromhex(block_hex)
    # pad_char is known per the challenge (e.g., ord('X') or whatever `c` is)
    ch, nonce = brute_nonce(key, ct_block, pad_char=ord('X'), block_idx=i)
    if ch:
        flag += ch
        print(f"Block {i}: '{ch}'  (nonce={nonce})")

print("Flag:", flag)

The per-block brute-force is 65536 AES operations. AES-ECB on modern hardware runs at hundreds of MB/s; 65536 × 16-byte blocks per flag character completes in milliseconds. The total runtime for a 30-character flag is under a second.

The exact format of the nonce within the 16-byte AES input depends on the implementation — you need to match eff.py’s layout exactly.

What I Learned#

Randomizing the nonce per block doesn’t help if the nonce space is too small. 2 bytes (65536 values) is trivially brute-forceable, especially when the plaintext has a known rigid structure that lets you verify candidates with near-zero false-positive rate. Nonces need to be long enough to make brute-force infeasible — typically 12 bytes for AES-GCM, not 2.
