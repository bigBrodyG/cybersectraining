Dan — ChaCha20 Stream Cipher Ticket Forgery

January 1, 2025 · 3 min · bigBrodyG
Table of Contents
  • The Challenge
  • Approach
  • Solution
  • What I Learned

The Challenge#

dan.py is a music ticket service built on ChaCha20. The server holds a list of songs; for each song you can request a ticket, which is a (nonce, ciphertext) pair where the ciphertext is ChaCha20(key, nonce, song_name). The key is fixed across all tickets. There’s a secret song — “Freddie Dredd - Cha Cha” (23 bytes) — and submitting a valid ticket for it unlocks the flag.

Approach#

ChaCha20 is a stream cipher: ciphertext = plaintext XOR keystream(key, nonce). The keystream depends on the key and the nonce; it doesn’t depend on the plaintext. If I have a ticket for a different song that uses the same nonce, I don’t get anything useful — different nonces mean different keystreams. But I don’t need the same nonce: I need a known plaintext encrypted under a nonce I can reuse or under the same keystream position.

The trick here is simpler: I can get the server to issue me a ticket for any song I want, including one with a known, long-enough name. Song 4 is “Jessica Jay - Chilly Cha Cha” — 28 bytes, which is longer than the 23-byte target. I request its ticket, which gives me (nonce, ct_4) where ct_4 = keystream(key, nonce) XOR song_4_name[0:28].

From that I recover the first 23 bytes of the keystream:

$$\text{keystream}[0:23] = ct_4[0:23] \oplus \text{“Jessica Jay - Chilly Cha Cha”}[0:23]$$

Then I forge the target ciphertext:

$$ct_\text{target} = \text{keystream}[0:23] \oplus \text{“Freddie Dredd - Cha Cha”}$$

And the ticket I submit is (nonce, ct_target). The server decrypts with the same key and nonce, recovers the song name, and it matches “Freddie Dredd - Cha Cha” — flag unlocked.

from pwn import *

# Connect and get ticket for song 4 ("Jessica Jay - Chilly Cha Cha")
r = remote('dan.chall.cyberchallenge.it', PORT)

known_song = b"Jessica Jay - Chilly Cha Cha"   # 28 bytes
target_song = b"Freddie Dredd - Cha Cha"        # 23 bytes

# Request ticket for song 4
r.sendlineafter(b'> ', b'1')        # get ticket
r.sendlineafter(b'song: ', b'4')
data = r.recvline().strip()

# Parse nonce and ciphertext from server response
# Format: nonce_hex:ct_hex  (adjust to actual format)
nonce_hex, ct_hex = data.split(b':')
nonce = bytes.fromhex(nonce_hex.decode())
ct_known = bytes.fromhex(ct_hex.decode())

# Recover keystream from known plaintext
keystream = bytes(ct_known[i] ^ known_song[i] for i in range(len(target_song)))

# Forge ciphertext for target song
ct_target = bytes(keystream[i] ^ target_song[i] for i in range(len(target_song)))

# Submit forged ticket
forged = nonce.hex() + ':' + ct_target.hex()
r.sendlineafter(b'> ', b'2')        # submit ticket
r.sendlineafter(b'ticket: ', forged.encode())
print(r.recvall().decode())

The key point is that I use the same nonce from the song 4 ticket when submitting the forged ticket. Since the keystream is (key, nonce)-deterministic, the server will produce the same keystream for that nonce, decrypt my forged ciphertext, and recover the target song name.

What I Learned#

Keystream reuse is the fundamental weakness of stream ciphers. If you can get the server to encrypt a known plaintext under the same (key, nonce) pair that protects your target, you own the keystream and can forge anything of the same or shorter length. Never reuse nonces with stream ciphers.
