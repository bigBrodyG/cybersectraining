Cry3 — AES-CBC Bit-Flip on Token Parser

March 1, 2025 · 3 min · bigBrodyG
Table of Contents
  • The Challenge
  • Approach
  • Solution
  • What I Learned

The Challenge#

secure_passphrase_generator.py presents a passphrase service. Internally, the server splits the flag into 4 chunks and places them at indices 0–3 of a word list; the remaining slots (4–23) contain filler words. When you authenticate, the server encrypts a token containing your username and four word indices using AES-CBC with a random server-side key. The token format after decryption looks something like:

1

username=alice;index0=4;index1=7;index2=12;index3=19

The recover function parses this — splitting on ;, then splitting each field on = — and returns words[index0], words[index1], words[index2], words[index3] as the passphrase. No integrity check, no HMAC.

Approach#

AES-CBC is not authenticated cipher text. Flipping a bit in ciphertext block C[i] causes a predictable single-bit flip in plaintext block P[i+1], at the exact same byte position, while completely corrupting P[i]. This is the classic bit-flip attack.

The token fields that matter are the four index values. If I can change index0=4 to index0=0, the server will look up words[0], which is flag_blocks[0] — the first quarter of the flag. Same for indices 1, 2, and 3. Since the username field comes first, corrupting it doesn’t matter; the parser still picks up the index fields from the second block onwards.

For each index field, I calculate which byte in the ciphertext I need to flip, figure out what the current value is versus what I want it to be, and XOR the corresponding ciphertext byte with current_value ^ target_value. The server decrypts, the bit-flip lands on the index digit, and the parse function reads the modified index.

No brute-force needed — the target bytes are known (they’re ASCII digits, so '4' is 0x34 and '0' is 0x30; the difference is just 0x04), and the offsets are deterministic once you know the token plaintext layout.

Solution#

There’s no standalone solve script for this one — the attack was worked out interactively against the live server. The conceptual procedure is:

  1. Authenticate and receive the encrypted token (IV + ciphertext as hex).
  2. Identify the byte positions of index0=4, index1=..., etc., within the plaintext. With a known username, the layout is fixed.
  3. For each index k (0–3): XOR ciphertext[offset_k] with ord(current_digit) ^ ord('k') to flip that digit to k.
  4. Send the modified token to the recover endpoint.
  5. The server decrypts and returns words[0], words[1], words[2], words[3] — the four flag fragments.
  6. Concatenate them.

The username block gets corrupted each time (that’s expected and acceptable since the parser ignores it after splitting on ;). You repeat the process up to four separate requests, one per flag fragment.

What I Learned#

AES-CBC without a MAC is not secure against chosen-ciphertext attacks. The bit-flip property is extremely well-known, but seeing a real service parse unsanitized fields out of a decrypted blob makes the danger concrete — and the attack trivially practical.
