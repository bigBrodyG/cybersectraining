Shuffled OX — Known-Plaintext Attack on Random Alphabet Substitution

March 1, 2026 · 3 min · bigBrodyG
Table of Contents
  • The Challenge
  • Approach
  • Solution
  • What I Learned

The Challenge#

shuffled.py encrypts the flag through three steps: base85 encode the raw flag bytes, apply a secret random permutation of the 256-byte alphabet as a substitution cipher, then hex encode the result. You connect to the server, receive the encrypted flag hex, and can then send arbitrary hex-encoded input to get back its encrypted form under the same permutation. The permutation is fixed for the session.

Approach#

This is a textbook known-plaintext attack. The server applies a fixed (but unknown) byte substitution. If I send input P, I get back E(P) where E is the substitution. I know P, I know E(P), so for every byte position I can build the mapping E[P[i]] = E(P)[i], or equivalently the reverse map E(P)[i] → P[i].

The flag is encrypted the same way. If I have a complete reverse map {encrypted_byte: plaintext_byte}, I can decrypt any ciphertext — including the flag.

The only subtlety is that the substitution is applied after base85 encoding. Base85 uses a 85-character alphabet (bytes 33–117 for standard base85). I need my known plaintext to be long enough that after base85 encoding, it covers — or at least contains — every byte that appears in the encrypted flag. Sending 2000 random bytes produces about 2500 base85 characters, which statistically covers most of the relevant byte values.

Once the reverse map is built from the known-plaintext exchange, decrypting the flag is a straight lookup: apply the map to each byte of the encrypted flag to recover the base85 encoding, then base85-decode to get the raw flag.

import os, base64
from pwn import *

r = remote('10.45.1.2', 2002)

# Receive the encrypted flag before sending anything
enc_flag_hex = r.recvline().decode().strip()
enc_flag = bytes.fromhex(enc_flag_hex)

# Send 2000 random bytes as known plaintext
inp = os.urandom(2000)
known_b85 = base64.b85encode(inp)

r.recvuntil(b'Hex message: ')
r.sendline(inp.hex().encode())
resp = r.recvline().decode().strip()
enc_inp_hex = resp.split('B85 message: ')[1]
enc_inp = bytes.fromhex(enc_inp_hex)

# Build reverse map: encrypted_byte → known_b85_byte
rev_map = {enc_inp[i]: known_b85[i] for i in range(len(known_b85))}

# Decrypt the flag
dec_b85 = bytes(rev_map[b] for b in enc_flag)
flag = base64.b85decode(dec_b85).decode()
print("Flag:", flag)

base64.b85encode uses Python’s standard base85 (RFC 1924-inspired, ASCII-safe). The server uses the same — confirmed by matching the exchange format. The reverse map dict is built in one pass over enc_inp and known_b85, zipped by index. Decrypting the flag is then a single list comprehension.

The one thing that can go wrong is if enc_flag contains a byte that wasn’t mapped in the known-plaintext exchange (i.e., a byte that my 2000 random bytes, after base85 encoding, never produced). With 2000 bytes → ~2500 base85 characters, the probability of missing any given byte value in the relevant alphabet is very low, but if it happens, the script throws a KeyError and you just re-run with a fresh random input.

What I Learned#

A substitution cipher with a known encryption oracle is trivially broken via a single known-plaintext query of sufficient length. The base85 intermediate layer doesn’t help — it reduces the effective alphabet, which actually makes the attack easier, not harder.
